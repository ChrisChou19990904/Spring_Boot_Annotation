前言

在社群平台分享連結時，常常會看到網址後面多了一串參數，例如：

https://外部網站.com/?fbclid=AJ82jsdOAsd...

這些參數是平台用來追蹤分享來源和使用者行為的工具，但也引發了許多關於隱私的疑問。本文將深入解析這些參數的作用、是否會暴露分享者身份，以及如何保護自己的隱私。

🔍 1. 原始問題：分享網址會不會暴露分享者？

我的問題：

「透過網址分享連結給他人時，為了避免發現原始貼文的分享者是誰，可以刪除 /? 之後的文字嗎？懂這個跟是不是工程師有關嗎？」

回答：
✔ 是的，多數情況下刪除 ? 後面的參數可以避免平台追蹤分享者的來源。
✔ 但這是一般「網路隱私常識」，不代表知道的人一定是工程師。

🧩 2. Query Parameters 是什麼？

? 後面的是查詢參數，用來傳遞額外資訊，例如：

https://example.com/article?id=123&utm_source=facebook

內容可能包含：

  • 廣告追蹤碼（UTM）
  • 社群分析參數
  • 分享者的資訊
  • 平台安全判斷用資料

📌 3. 為什麼 Facebook 會加一長串參數？

因為我處在「已登入」狀態，平台需要：

  • 追蹤我來自哪裡（追蹤碼：fbclid）
  • 確認我有沒有從可疑來源跳轉
  • 判斷分享來源（群組？動態牆？粉專？）

因此會看到像：

https://外部網站.com/?fbclid=AJ82jsdOAsd...

📌 4. 為什麼 GitHub Pages 不會有一串參數？

我的 GitHub 靜態網站是：

  • 沒有登入系統
  • 沒有追蹤功能
  • 沒有廣告/分析系統

所以它不會自動加參數，網址會保持：

https://我的網站.github.io/page.html

📊 5. 各平台常見追蹤參數列表

Facebook

fbclid=
ref=
__tn__=
__cft__=

Instagram

igshid=
utm_source=ig
utm_medium=copy_link

LINE

utm_source=line
utm_campaign=share
utm_medium=messenger

YouTube

pp=
si=
feature=
t=   ← 保留是時間軸

以上參數全部都可以刪除，不會影響目標網站使用。

⚠ 6. 哪些參數可能會暴露我的身份？（注意）

  • 推薦碼、會員碼（如 ref=ABC123）
  • 錯誤實作的 Token / Session（少見但危險）
  • 個人分享 ID（Dcard、Medium 部分功能）

當你遇到上述參數時請記得刪掉它們。

🛡 7. 如何「匿名」分享連結

方法 1：刪掉 ? 後方所有參數

https://example.com/article

方法 2：使用網址淨化工具

  • removeutm.com
  • clearer.link

方法 3：使用瀏覽器插件（Redirect Cleaner）

🏁 8. 總結

我的理解完全正確：

  • Facebook 這類平台會加入追蹤參數，所以網址變長。
  • 我的 GitHub 靜態網站沒有登入系統 → 不會加參數。
  • 大部分參數只是用來追蹤，不是功能必需。
  • 刪掉通常不會影響網站正常運作。

我有的不是「工程師技巧」而是「網路敏感度」，這是很重要的能力。

🧠 9. 進階問題：為什麼沒有 ? 也會顯示分享者資訊？

我的新問題：

「朋友分享一段 Facebook 影片到群組，網址裡有 share 字樣， 但沒有 ? 及後續參數。 可是點進去卻會顯示是透過他的帳號分享的，為什麼？」

核心答案：

  • 追蹤資訊不一定放在 ? 後面。
  • 有可能直接寫在網址「路徑（Path）」裡。
  • 也可能透過 Cookie + 登入狀態動態顯示。

📐 10. Facebook 連結結構完整解析

一個網址其實包含三種可能的資料來源：

https://facebook.com/username/videos/123456789?fbclid=xxxx
│
├── Path（路徑） → username / videos / 123456789
│
├── Query（?後面） → fbclid=xxxx
│
└── Cookie（看不到） → 登入狀態 / 使用者識別

三種資料儲存位置說明：

  • Path：可能包含分享者 ID 或貼文 ID。
  • Query string：通常是廣告或來源追蹤。
  • Cookie：完全不顯示在網址，但會告訴伺服器你是誰。

即使沒有 ?，只要 Path 裡含有使用者識別碼， 伺服器仍然可以判斷分享來源。

📡 11. 分享行為的資料流示意

[朋友點擊「分享」]
        ↓
Facebook 產生一筆新貼文（屬於他的帳號）
        ↓
系統產生新的貼文 ID
        ↓
網址指向：
facebook.com/他的帳號/videos/貼文ID
        ↓
你點擊該網址
        ↓
伺服器讀取貼文ID → 查資料庫 → 顯示「某某分享了這段影片」

重點：那個連結可能不是「原始影片連結」，而是「他分享後產生的新貼文」。

🍪 12. Cookie / Session / URL 三者關係圖

情境一：你是登入狀態

你的瀏覽器
  ├── URL
  ├── Cookie（包含你的登入資訊）
  └── Session ID
        ↓
Facebook 伺服器知道：
  - 你是誰
  - 你和分享者是否好友
  - 你是否在同一群組
        ↓
動態顯示分享者資訊

情境二：無痕模式 / 未登入

沒有 Cookie
        ↓
伺服器只能根據網址 Path 判斷
        ↓
如果 Path 沒有分享者 ID
→ 就不會顯示分享者資訊

這就是為什麼「用無痕模式測試」可以判斷 分享資訊是來自 URL 還是來自登入狀態。

🏁 13. 進階總結

  • 沒有 ? 不代表沒有追蹤。
  • 現代網站常把識別資訊寫在 Path。
  • 登入狀態會改變頁面顯示內容。
  • 社群平台通常是「動態渲染頁面」，不是單純靜態網址。

因此：
網址是否包含 ? 並不是判斷是否暴露分享者的唯一依據。

🚫 14. 那個 share 連結能不能避免暴露分享者？

我的問題：

「所以那個 share 的部分就完全不能避免暴露分享者資訊嗎？」

回答重點：

  • 如果那個連結本身就是「他分享後產生的新貼文」，幾乎無法匿名。
  • 因為那不是追蹤碼，而是一則屬於他帳號的貼文。
  • 刪除 share 並不會改變貼文歸屬。

兩種可能情況：

① 那是分享後的新貼文

facebook.com/他的帳號/posts/123456789
facebook.com/share/v/123456789

這種網址指向的是「他的貼文」，不是原始影片。 因此一定會顯示分享者。

② 只是登入狀態造成的顯示

如果在無痕模式（未登入狀態）下打開該網址， 不再顯示分享者資訊， 那表示該顯示是由 Cookie 與登入狀態動態產生， 而不是網址本身暴露。

如何驗證？

  1. 複製該連結
  2. 開啟瀏覽器無痕視窗
  3. 不要登入 Facebook
  4. 貼上網址並觀察是否仍顯示分享者

如果仍顯示 → 那是貼文本體（無法匿名）
如果不顯示 → 那是登入狀態造成（可避免）

⚡ 15. 3 秒匿名判斷法

當你看到一個社群連結時，可以用以下「3 秒檢查法」 快速判斷是否可能暴露分享者資訊。

✅ 第 1 秒：找 ?

https://example.com/video/123456?fbclid=xxxx&utm_source=facebook

? 後面的通常是追蹤參數（Query String）。
多數情況可以刪除，不影響內容本體。

  • fbclid=
  • utm_source=
  • utm_medium=
  • ref=

👉 有 ? 多半只是追蹤碼。

✅ 第 2 秒：看路徑是否包含帳號或群組

facebook.com/某人名字/posts/123456
facebook.com/groups/群組ID/permalink/123456
facebook.com/profile.php?id=數字

如果網址路徑中包含：

  • 使用者名稱
  • profile.php?id=
  • groups/群組ID

🚨 代表這是某個帳號或群組底下的貼文。
這不是追蹤碼，而是內容本體的歸屬。

✅ 第 3 秒：無痕模式驗證

  1. 複製該網址
  2. 開啟瀏覽器無痕視窗
  3. 不要登入社群平台
  4. 貼上網址查看是否仍顯示分享者

結果判讀：

  • 仍顯示分享者 → 那是貼文本體，無法匿名。
  • 不再顯示 → 那是登入狀態與 Cookie 造成的顯示。

🎯 一句話速記

有 ? 多半是追蹤。
有人名或群組在路徑 → 是貼文歸屬。
無痕模式是最終驗證方法。

🔬 16. 進階工程檢查法（DevTools / Network / GraphQL）

如果想確認一個連結到底是在讀「某個使用者的貼文」，還是單純影片 ID，可以用瀏覽器開發者工具來檢查。

🪜 步驟一：開啟開發者工具

  • Windows：按 F12
  • Mac：Command + Option + I
  • 或右鍵 → 檢查

切換到 Network（網路） 分頁，重新整理頁面。

🪜 步驟二：觀察 request

  • 注意 request URL 中的 graphql / video / story / profile / feed
  • 點開 request，檢查回傳的 Response
  • 搜尋 owner / author / profile / actor
"owner": {
  "id": "1000xxxxxxx",
  "name": "某某"
}

如果看到類似 owner 或 profile 的資訊 → 表示該連結指向某個帳號的貼文。

🧠 判讀邏輯

  • 只有 video_id → 原始影片，無帳號歸屬
  • 出現 story_id + owner_id → 分享貼文，屬於某帳號

⚠ 注意事項

  • 登入狀態會顯示更多個人化資料
  • 無痕模式 / 未登入最能看出 URL 本身資訊與登入狀態的差異

這種方法讓你直接從網路請求看清楚，分享者資訊到底是不是內嵌在內容本體裡，而不是靠網址參數控制。

🎯 快速工程師判斷法

  • Network 裡有 owner → 帳號貼文，無法匿名
  • 只有 video → 單純影片，可匿名
  • 刪掉 query string 對貼文歸屬沒影響

📡 17. GraphQL / URL / Cookie / Session 三者關係圖

現代社群平台（以 Facebook 為例）並不是單純靠網址顯示內容，而是結合：

  • URL 路徑：貼文 ID / 使用者 ID / 群組 ID
  • Query string：追蹤來源 / 廣告參數（可刪）
  • Cookie / Session：登入狀態、個人化資訊
  • GraphQL API：動態拉取資料並渲染頁面

🔗 資料流示意

[使用者點擊分享連結]
        │
        ▼
URL（Path + Query）送到伺服器
        │
        ▼
伺服器檢查 Cookie / Session
        │
        ├── 有登入 → 回傳個人化資料（如：好友、群組權限）
        └── 無登入 → 只回傳公開內容
        │
        ▼
GraphQL / API 回傳 JSON
        │
        ▼
前端動態渲染頁面
        │
        ├── 顯示「某某分享」 → URL 指向貼文本體 + API 回傳 owner
        └── 只顯示影片 → URL 只是影片 ID，無 owner

重點：即使刪掉 ?utm... 參數，分享者資訊仍可能透過 URL 路徑 + API + Cookie 顯示。

🎯 小結

  • 追蹤參數可刪，但貼文本體屬於帳號 → 無法完全匿名
  • 登入狀態影響顯示內容，無痕模式可驗證
  • GraphQL API 是現代平台用來動態拉資料並渲染頁面的核心機制
