學程式的階段，以及「輸入」在不同階段的價值

前言

初學程式時，常會接觸到像 input()（Python）、 Scanner / next()（Java）這類 「只能在本地輸入資料」的工具。

當開始真正開發前後端系統後，這些工具幾乎不再出現， 於是自然會產生疑問：

既然實務系統不太用它們，那它們存在的意義是什麼？

第一階段：學會「如何寫程式」

目標

  • 建立程式邏輯思維
  • 理解控制流程（if / loop / function / container）
  • 訓練手刻程式碼的基本能力

常見工具

  • input()
  • Scanner、next()
  • 標準輸入 / 輸出（CLI）

在這個階段，輸入工具的角色是：

用最小成本，讓初學者理解「程式可以接收外部資料並做出反應」。

這時候的重點不是產品，而是「程式腦」的養成。

第二階段：真的在做「可用的系統」

目標

  • 理解並使用框架（Web / Backend / Frontend）
  • 設計模組與系統架構
  • 連接資料庫、處理 API
  • 部署（Docker / 雲端）
  • 撰寫與管理單元測試、相依套件
  • 善用 AI 生成 boilerplate 程式碼

輸入資料的來源

  • HTTP Request（表單、Query、JSON）
  • 前端 UI 操作
  • 資料庫
  • 訊息佇列或外部服務
在這個階段，輸入不再來自鍵盤， 而是來自「系統之間的資料流動」。

一個關鍵觀念：輸入是可替換的

核心邏輯應該與輸入方式解耦，例如：

def search_books(keyword): # 核心商業邏輯 pass

不同階段，只是換了輸入來源：

  • CLI：keyword = input()
  • Web：keyword = request.args["q"]
  • API：keyword = json["keyword"]
程式真正的價值在「處理資料的邏輯」，而不是資料從哪裡來。

為什麼在第二階段幾乎不用 input / Scanner？

  • 它們不是使用者真正互動的介面
  • 無法代表實際系統的資料流
  • 系統重點已轉向架構、可維護性與部署

但它們並沒有完全消失，而是退居為：

  • CLI 工具
  • 管理腳本
  • 測試或資料初始化輔助程式

總結

第一階段學的是「怎麼寫程式」，
第二階段學的是「怎麼把程式變成系統」。

input() 與 Scanner 的價值， 不在於未來會不會用它們做產品， 而在於它們幫助你跨出學會思考程式的第一步。
